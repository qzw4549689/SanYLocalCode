# 资信附件上传功能配置说明

> 本文档说明如何在 `mcs_credit_record`（客户信用评估记录表）表单中嵌入三一通用上传组件 `mcs_/CommonCore/Html/Uploader.html`，实现客户资信附件的上传与管理。

---

## 一、功能概述

原 PRD 中未对资信附件模块做专门描述，但项目已创建 `mcs_customer_file`（客户资信附件表）实体。本方案复用三一 D365 平台已有的通用上传组件，在评估记录表单中展示并上传**当前客户（account）**的资信附件。

### 设计要点

- **附件归属**：资信附件属于**客户（account）**，而不是某一条评估记录（`mcs_credit_record`）。因此多个评估记录可以共享同一个客户的附件。
- **Uploader 参数**：通用上传组件通过 `entityName` + `entityId` 定位附件所有者。
  - `entityName` = 业务实体名 `account`
  - `entityId` = 当前评估记录所选客户 `mcs_accountid`
- **文件存储**：上传组件后端会创建 `mcs_customer_file` 记录，并通过 `mcs_accountid` 关联到客户。
- **类型映射**：`UploadFileTypeMapping` 的 key 使用业务实体名 `account`，而不是文件实体名 `mcs_customer_file`。

## 二、D365 中手动配置步骤

### 2.1 配置 UploadFileTypeMapping

**入口**：`Sany CRM Admin` 应用 → `System Setting` → `System Config`

在 `ms_systemconfiguration` 中新增或更新名为 `UploadFileTypeMapping` 的记录，添加如下配置：

```json
{
  "account": [
    {"Type": "001", "Max": 10},
    {"Type": "002", "Max": 10},
    {"Type": "003", "Max": 10},
    {"Type": "004", "Max": 10},
    {"Type": "005", "Max": 10},
    {"Type": "006", "Max": 10},
    {"Type": "007", "Max": 10},
    {"Type": "008", "Max": 10},
    {"Type": "099", "Max": 20}
  ]
}
```

> **重要**：key 必须是业务实体名 `account`，不能是文件实体名 `mcs_customer_file`。通用上传组件后端会根据 `entityName=account` 查找此映射。

**Type 与 `mcs_filetype` 选项集映射**：

| Uploader Type | mcs_filetype 值 | 含义 |
|---------------|----------------|------|
| 001 | 1 | 客户基础资质 |
| 002 | 2 | 客户资信报告 |
| 003 | 3 | 重点尽调报告 |
| 004 | 4 | 客户买方财务报表 |
| 005 | 5 | 客户担保方财务报表 |
| 006 | 6 | 纳税证明 |
| 007 | 7 | 担保函或履约保函 |
| 008 | 8 | 行业排名 |
| 099 | 99 | 其他 |

> 如果通用上传组件在创建 `mcs_customer_file` 记录时传入的是 Uploader Type（如 `"001"`），则需要在 Plugin 或 JS 中做映射转换。当前 `CustomerFileAutoNumberPlugin` 不处理类型映射，如需转换请补充逻辑。

### 2.2 配置多语言文件

**文件位置**：D365 解决方案 `SanyD365.D365WebResource` → WebResource → `ms_languagefile_xxxx.json`

在对应语言文件中添加 Key（以中文 2052 为例）：

```json
{
  "account_filetype_001": "客户基础资质",
  "account_filetype_002": "客户资信报告",
  "account_filetype_003": "重点尽调报告",
  "account_filetype_004": "客户买方财务报表",
  "account_filetype_005": "客户担保方财务报表",
  "account_filetype_006": "纳税证明",
  "account_filetype_007": "担保函或履约保函",
  "account_filetype_008": "行业排名",
  "account_filetype_099": "其他"
}
```

```json
{
  "account_filetype_001": "Basic Qualification",
  "account_filetype_002": "Credit Report",
  "account_filetype_003": "Due Diligence Report",
  "account_filetype_004": "Buyer Financial Statement",
```

## 三、前端传参说明

### 3.1 窗体 XML

`mcs_credit_record` 主窗体中 Uploader WebResource 的 XML 示例：

```xml
<control id="mcs_credit_record_uploader" classid="{9FDF5F91-88E1-47cd-9CAE-4C7186A64CBD}">
  <parameters>
    <Url>mcs_/CommonCore/Html/Uploader.html</Url>
    <PassParameters>true</PassParameters>
  </parameters>
</control>
```

> 不需要在窗体 XML 中写死 `Data` 参数。`entityName` / `entityId` / `fullFileTypes` 由表单 JS 动态传入，以保证切换客户时附件列表自动刷新。

### 3.2 JS 动态传参

`mcs_credit_record.js` 中的 `initAttachmentTab` 方法会：

1. 读取当前表单的 `mcs_accountid` 字段。
2. 如果已选择客户，则拼接 Uploader URL 参数：

```json
{
  "entityName": "account",
  "entityId": "<accountid>",
  "fullFileTypes": "001,002,003,004,005,006,007,008,099"
}
```

3. 将 JSON 字符串经过 `encodeURIComponent(JSON.stringify(data))` 后附加到 iframe 的 `src` 上。
4. 如果未选择客户，则清空 `data` 参数，避免 Uploader 误将当前评估记录作为附件所有者。

### 3.3 客户变更联动

`mcs_accountid` 字段的 `onChange` 事件会触发 `initAttachmentTab`，因此切换客户后附件页签会自动刷新为对应客户的附件列表。

## 五、测试要点

1. **基础功能**：在 `mcs_credit_record` 表单中选择客户后，切换到【附件】页签，能正常显示上传组件，不再报 "Loading data failed"。
2. **参数正确性**：在浏览器开发者工具中检查 Uploader iframe 的 `src`，确认 `data` 参数中包含 `entityName=account` 和当前 `accountid`。
3. **切换客户**：修改 `mcs_accountid` 后，附件页签应刷新为新客户的附件列表。
4. **上传后记录**：上传文件后，检查是否创建了 `mcs_customer_file` 记录，并验证：
   - `mcs_fileid` 格式为 `ATTYYYYMMDD####`
   - `mcs_filedate` 默认填充当前日期
   - `mcs_accountid` 正确关联到当前客户
   - `mcs_filename` 自动填充原文件名
   - `mcs_filetype` 与所选分类一致
5. **类型过滤**：确认附件类型下拉框只显示 001~008、099 这 9 种类型。
6. **多语言**：切换语言后，附件类型显示正确。
7. **数量限制**：测试每类文件最大上传数量是否生效。

## 八、复用：为其他实体添加 Uploader Tab

已在 `Code/Tools/MetadataTool` 中封装通用命令 `add-uploader-tab`，可以为任意实体的主窗体添加一个包含 Uploader HTML WebResource 的 Tab。

### 用法

```bash
cd Code/Tools/MetadataTool
export D365_URL="https://dev1.crm5.dynamics.com"

dotnet run add-uploader-tab <实体名> <Tab 名称> <WebResource URL>
```

### 示例

```bash
# 为 mcs_credit_record 添加 Attachments Tab
dotnet run add-uploader-tab mcs_credit_record Attachments mcs_/CommonCore/Html/Uploader.html
```

### 命令行为

1. 查询指定实体的主窗体（`type = 2`）
2. 检查是否已存在同名 Tab，避免重复添加
3. 在 `</tabs>` 前插入新的 Tab XML
4. 更新 D365 中的 `systemform` 记录
5. **不会自动发布实体**，需手动在 D365 中发布窗体或执行 `dotnet run publish <实体名>`

### 注意事项

- 该命令直接修改 D365 中的窗体 XML，不依赖 solution 导入
- 执行前请确保 DEV 环境没有正在进行的 solution 导入/发布操作
- 如果 Tab 已存在，命令会跳过
- 附件的业务上下文（如 `entityName` / `entityId`）需由对应实体的表单 JS 动态传入

## 九、常见错误排查

### 9.1 Uploader 报 "Loading data failed"

可能原因：

1. `UploadFileTypeMapping` 中缺少当前 `entityName` 的配置。请确认 key 为 `account`。
2. `entityId` 为空或不是有效的 GUID。请确认已选择客户且 JS 正确传参。
3. `Data` 参数格式错误。Uploader 期望 `data` 参数值为 JSON 字符串，而不是 `entityName=xxx` 的键值对。
4. DEV 环境发布冲突或网络超时。稍后重试或检查 D365 后台作业。

### 9.2 附件类型显示不全或包含未配置类型

通用上传组件对 `account` 实体可能返回默认类型 000~012。如需只显示业务需要的类型，请在 `data` 参数中传入 `fullFileTypes` 进行过滤。
