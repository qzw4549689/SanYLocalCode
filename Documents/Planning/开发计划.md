#### 4.2.4 信用分计算完整流程（2026-06-08更新）

**触发条件**：
```
实体: mcs_credit_record
消息: Update
阶段: PostOperation (40)
条件: mcs_status 值变更为 13 (信用分计算)
```

**执行流程（4个阶段）**：

**阶段1: 获取评分卡类型**
```
步骤1: 查询客户Account属性 (mcs_accountcategory, mcs_accountlevel, mcs_accounttype)
步骤2: 查询销售订单数量 (≥1=老客户, 0=新客户)
步骤3: 匹配评分卡类型 (7种类型, 详见3.4.1评分卡匹配规则)
步骤4: 校验评分卡配置是否存在 (mcs_credit_scoringcard表中categoryId对应记录)
```

**阶段2: 获取评分卡配置**
```
查询实体: mcs_credit_scoringcard
条件: mcs_categoryid = 匹配的categoryId
字段: mcs_credititem(Lookup→mcs_credit_items), mcs_itemname, mcs_datatype,
      mcs_listvalue, mcs_minvalue, mcs_maxvalue, mcs_weight
关键字段类型注意:
  - mcs_credititem: EntityReference (Lookup → mcs_credit_items)
  - mcs_listvalue: EntityReference (Lookup → 定性枚举值), 非String
  - mcs_datatype: OptionSetValue (100000000=定量, 100000001=定性)
  - mcs_minvalue/mcs_maxvalue: Decimal
  - mcs_weight: Int32
```

**阶段3: 获取客户信用标签（复核值优先，原始值回退）**
```
查询实体: mcs_customer_tag
条件: mcs_credit_record = 当前评估记录ID AND mcs_active = true
字段: mcs_credit_item(Lookup), mcs_datatype,
      mcs_itemintvalue2(复核定量), mcs_itemtxtvalue2(复核定性), mcs_itemvalue2(复核通用),
      mcs_itemintvalue1(原始定量), mcs_itemtxtvalue1(原始定性), mcs_itemvalue1(原始通用)

取值优先级（关键业务规则）:
  1. 优先读取复核值 (value2)
     - 定量: mcs_itemintvalue2 → 如为空则读 mcs_itemvalue2
     - 定性: mcs_itemtxtvalue2 → 如为空则读 mcs_itemvalue2
  2. 复核值为空时，回退读取原始值 (value1)
     - 定量: mcs_itemintvalue1 → 如为空则读 mcs_itemvalue1
     - 定性: mcs_itemtxtvalue1 → 如为空则读 mcs_itemvalue1
  3. 仍为空则视为缺失值
     - 定量默认值: -1 (表示缺失, 得0分)
     - 定性默认值: "O" (表示缺失, 得0分)

业务背景:
  - value1 = Coface数据集成Plugin写入的原始值（API返回数据）
  - value2 = 人工复核阶段用户可修改的复核值
  - 用户未复核时(value2为空)，自动使用原始值(value1)计算
```

**阶段4: 逐项计算得分**
```
遍历评分卡配置项:
  1. 按评分项目编码(mcs_credit_items.mcs_credit_itemsno)匹配标签记录
  2. 数据类型判断: isQuantitative = (dataType == 1 || dataType == 100000000)

定量指标得分规则:
  - 缺失值(-1): 0分
  - 值在 [MinValue, MaxValue) 范围内: 得Weight分 (左闭右开区间)
  - 值不在范围内: 0分

定性指标得分规则:
  - 缺失值("O")或空值: 0分
  - ListValue为空/未配置: 默认给Weight分（无匹配配置时兜底）
  - value 与 ListValue 忽略大小写匹配: 得Weight分
  - 不匹配: 0分

总分 = 所有指标得分之和
```

**阶段5: 更新评估记录**
```
更新字段:
  - mcs_creditscore = (decimal)totalScore
  - mcs_scoredate = DateTime.Now
  - mcs_status = 14 (审核申请)

注意: BPF状态同步由BpfStageSyncPlugin统一处理，CreditScorePlugin不直接操作BPF

#### 5.4.2 执行流程（5个阶段）

**阶段1: 获取URBA360数据（9个指标）**
```

  1. POST /authentication/v1/token → 获取Token
  2. GET  /dataapi-v1/urba360/monitorings/orders → 查询URBA订单列表
  3. 【订单状态检查】→ 判断订单是否存在及状态
  4. POST /authentication/v1/token → 重新获取Token
  5. GET  /dataapi-v1/urba360/content → 获取URBA JSON内容（~190KB，仅当订单就绪时）

订单状态判断逻辑:
  - 订单列表为空 → 状态=NotFound，提示"该客户尚未下单"
  - 订单存在且status=ready/partially_ready → 状态=Ready，继续获取内容
  - 订单存在但status为空/不存在，且有订单ID → 兼容处理，视为就绪（Coface某些环境不返回status字段）
  - 订单存在但status≠ready/partially_ready且不为空 → 状态=NotReady，提示"订单未就绪，需等待"

指标解析（Urba360Parser）:
  - 外部评级: productDetails.score[].debtorRiskValue
  - 迟付指数: productDetails.latePaymentIndex[].value
  - 国别风险: productDetails.countryRiskAssessment[].countryRiskValue
  - 行业风险: productDetails.sectorRiskAssessment[].countryRiskValue
  - 行业属性: companyGeneralInformation.naceCodes[].code（提取前2位Division→映射三一行业→多值去重拼接）
  - 净资产: productDetails.financials.balanceSheet.balanceSheetItems[].indicators[]（匹配Net Assets/Net Worth/Equity/Own funds，应用dimension换算和货币转换）
  - 资产负债率: productDetails.financials.ratios[].name="Debt ratio"
  - 流动比率: productDetails.financials.ratios[].name="Current Ratio"
  - 净利润率: productDetails.financials.ratios[].name="Net Profit Margin"/"ROS"
```

**阶段2: 获取Full Report数据（3个指标）**
```

  6. POST /authentication/v1/token → 获取Token

  - 订单存在但status为空/不存在，且有订单ID → 兼容处理，视为就绪（Coface某些环境不返回status字段）

#### 5.4.3 订单状态处理策略

| 订单类型 | 状态 | 处理方式 | API消息 |
|---------|------|---------|---------|
| URBA360 | NotFound（未下单） | 9个指标填充缺失值 | 包含"URBA360:未下单"警告 |
| URBA360 | NotReady（未就绪） | 9个指标填充缺失值 | 包含"URBA360:订单未就绪"警告 |
| URBA360 | Ready（就绪） | 正常获取并解析9个指标 | 无警告 |
| URBA360 | EmptyStatus（status为空） | 正常获取并解析9个指标（兼容处理） | 无警告 |
| Full Report | NotFound（未下单） | 3个指标填充缺失值 | 包含"Report:未下单"警告 |
| Full Report | NotReady（未就绪） | 3个指标填充缺失值 | 包含"Report:订单未就绪"警告 |
| Full Report | Ready（就绪） | 正常获取并解析3个指标 | 无警告 |
| Full Report | EmptyStatus（status为空） | 正常获取并解析3个指标（兼容处理） | 无警告 |

> **业务规则**: 未下单只提示"没有订单"；已下单且订单ready才取数。订单未就绪时填充缺失值，不影响整体流程（仍标记SUCCESS，但API消息包含警告）。

#### 5.4.4 异常处理策略

| 场景 | 处理方式 |
|------|---------|
| 无URBA360订单 | 9个指标填充缺失值，继续执行Report |
| 无Full Report订单 | 3个指标填充缺失值，整体标记SUCCESS |
| API调用失败 | catch异常，填充缺失值，继续执行 |
| JSON解析失败 | catch异常，该指标填充缺失值，不影响其他指标 |
| 字段值超出范围 | 截断为99,999,999,999.99 |
| 字段值小于0 | 按缺失值处理（value1="N/A"） |
| 评分项目不存在 | 跳过该指标，继续处理其他 |
| Token获取失败 | 抛出异常，中断执行 |
| 事务冲突 | 合并为单次Update，移除中间Update |

#### 5.4.5 关键设计决策

| 决策 | 说明 |
|------|------|
| 动态评分卡配置 | 根据评分卡配置动态创建标签（非固定12个指标） |

| 数据类型自动转换 | 评分卡datatype(100000000/100000001) → 标签datatype(1/2) |

| Token每次重新获取 | 每次API调用前重新获取Token（不缓存），简化逻辑 |

**审批触发时机：**
- 信用分计算完成后
- 用户点击【下一步】进入【审核申请】阶段
- 系统自动调用BPP提交审批接口

**审批流程：**
```
信用分计算完成 → 点击【下一步】→ 进入审核申请 → 调用BPP提交接口 → BPP审批中 → 审批结果回传
```

**审批结果处理：**
| 结果 | 状态更新 | 后续操作 |
|------|---------|---------|
| 审批通过 | mcs_status = 15（审批通过） | 1. BppCallbackPlugin更新mcs_status=15, mcs_active=true<br>2. 更新Account表信用分/等级/有效状态 |
| 审批驳回 | mcs_status = 12（人工复核） | 1. BppCallbackPlugin更新mcs_status=12<br>2. mcs_bpprejectreason记录驳回原因<br>3. 支持修改后重新提交到状态14 |

**涉及字段（mcs_credit_record）：**
| 字段 | 用途 |
|------|------|
| mcs_bppid | BPP工作流ID |
| mcs_bppstatus | BPP审批状态 |
| mcs_bppappriver | 当前审批人 |
| mcs_approvedate | BPP审批完成日期 |
| mcs_bpperrormsg | BPP错误信息 |
| mcs_bpprejectreason | BPP驳回原因 |

**技术方案：**
1. **提交审批**：Plugin/Flow 调用 BPP API 提交审批申请
2. **状态查询**：定时任务或手动刷新查询BPP审批状态

## 20. 功能概述

客户信用画像模块是信用评估流程的**数据消费层**，承接 Phase 1~6 评估流程产生的数据，在客户管理模块中以页签形式展示客户信用综合信息，并支持被线索、配置报价、合同等业务环节独立调用。

**核心原则**：画像本身不复用新的评估流程，而是读取已有数据（account + mcs_customer_tag + 授信额度）。

## 21. 功能点详细设计

**信用基础标签和评分**:
```sql
-- 逻辑伪代码
SELECT mcs_group, mcs_itemname, mcs_itemdesc, mcs_itemvalue2, mcs_scorevalue
FROM mcs_customer_tag
WHERE mcs_accountid = [当前客户]
  AND mcs_group = 5        -- 综合指标（PRD要求）
  AND mcs_isscore = 1      -- 参与评分
ORDER BY mcs_group, modifiedon DESC
-- 同一指标取最新记录
```
> ⚠️ **注意**：PRD 要求过滤条件为 `评分项目分类 = 综合指标(5) AND 是否评分 = 1`。若业务需要展示全部分类指标，可扩展为展示 1-5 全部分类。

**当前采购要求**:
> 依赖 LTC 合同模块数据，当前阶段标注为"依赖外部系统"。建议通过在手订单合同视图或自定义 API 获取：
- 统计时点 t 前审批完成/备货状态的合同
- 金额 = SUM(合同金额 - 开票金额)
- 首付款比例 = MIN(各合同首付款比例)
- 账期 = MAX(各合同账期天数)
- 授信额度 = 对应授信额度记录

**重点尽调专家意见**:
> 依赖重点尽调模块（尚未开发）。当前阶段：当 `mcs_isdd = 1` 时显示区域框架，内容标注"待重点尽调模块提供"；当 `= 0` 时隐藏该区域。

### 21.4 F7.4 客户飞轮图形化

### 21.5 F7.5 内部接口设计

#### 21.5.1 客户信用标签评分查询 API

**实现方式**: D365 Custom API Action 或 ASP.NET WebResource HTTP Handler（推荐 Custom API，可复用现有服务上下文）

**请求参数**:
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| mcs_buyercode | string | 是 | 客户编码（account.accountnumber 或 mcs_customermasterdata 客户编码） |

**响应参数**:
| 字段 | 类型 | 说明 |
|------|------|------|
| status | string | "1"=成功, "0"=错误 |
| message | string | 错误详情 |
| data | array | 标签评分记录列表 |

**data 列表项**:
| 字段 | 类型 | 说明 |
|------|------|------|
| mcs_group | string | 评分项目分类（1客户实力/2客户财务/3宏观市场/4历史交易/5综合指标） |
| mcs_source | string | 数据来源（内部/外部/人工补入） |
| mcs_itemname | string | 评分项目名称 |
| mcs_scorevalue | number | 得分值 |

**后端查询逻辑**:
```csharp
// 伪代码
1. 根据 buyercode 查找 accountid
2. 查询该客户最新有效的 mcs_credit_record（mcs_active = 1，modifiedon 最大）
3. 查询该评估记录关联的 mcs_customer_tag
4. 过滤：mcs_isscore = 1
5. 返回：group, source, itemname, scorevalue
```

### 21.6 F7.6 画像权限控制

#### 21.6.1 权限矩阵

| 角色 | 数据范围 | 操作 |
|------|---------|------|
| 营销人员 | 本人负责的客户 | 查询 |
| 营销管理人员 | 本部门及下级部门的客户 | 查询 |
| 事业部风控人员 | 本部门及下级部门的客户 | 查询 |
| 总部风控人员 | 全集团客户 | 查询 |

#### 21.6.2 技术实现

- **D365 原生权限层**: 通过角色 + 业务部门层级 + 记录级安全性控制 account 实体访问
- **画像页面 JS 层**: 页面加载时调用 `Xrm.WebApi.retrieveMultipleRecords("systemuserroles", ...)` 判断当前用户角色，结合 `account.owninguser` / `owningbusinessunit` 做二次校验
- **飞轮独立页面**: 由于独立页面无法直接获取 Xrm 上下文，权限控制通过 API 层实现（API 内部校验当前用户是否有权访问该客户数据）
