# 评分卡类型匹配规则说明

> 创建日期: 2026-06-06
> 说明: 信用分计算Plugin的评分卡类型匹配逻辑，基于D365实际选项集值
> 关联代码: `/Code/Customizations/Plugins/CreditScore/Plugin/CreditScorePlugin.cs`

## 评分卡类型概览

PRD定义共 **7种评分卡**：

| 评分卡类型 | categoryId | 适用客户 | 说明 |
|-----------|-----------|---------|------|
| SA级老客户 | 1 | 直销大客户 + 老客户 | S级或A级直销客户，有历史交易 |
| SA级新客户 | 2 | 直销大客户 + 新客户 | S级或A级直销客户，无历史交易 |
| BC级老客户 | 3 | 直销普通客户 + 老客户 | B级或C级直销客户，有历史交易 |
| BC级新客户 | 4 | 直销普通客户 + 新客户 | B级或C级直销客户，无历史交易 |
| 个人客户 | 5 | 个人客户(I级) | 个人客户 |
| 老经销商 | 6 | 经销商 + 老客户 | 有历史交易的经销商 |
| 新经销商 | 7 | 经销商 + 新客户 | 无历史交易的经销商 |

## 匹配因子

评分卡类型由 **3个输入因子** 决定：

### 因子1：新老客户

| 判断方式 | 数据来源 | 结果 |
|---------|---------|------|
| 老客户 | `salesorder` 表中存在该客户的订单记录 | `isOldCustomer = true` |
| 新客户 | `salesorder` 表中无该客户订单记录 | `isOldCustomer = false` |

### 因子2：是否经销商

| D365字段 | 值 | 含义 | 结果 |
|---------|-----|------|------|
| `mcs_accountcategory` | **10** | Official Dealer（正式经销商）| `isDealer = true` |
| | **90** | Prospective Dealer（意向经销商）| `isDealer = true` |
| | 其他 | 直销客户 | `isDealer = false` |

### 因子3：直销客户分级

| D365字段 | 值 | 标签 | PRD映射 | 客户级别 |
|---------|-----|------|---------|---------|
| `mcs_accountlevel` | **4** | Diamond | **S级** | 大客户 |
| | **3** | Gold | **A级** | 大客户 |
| | **2** | Silver | **B级** | 普通客户 |
| | **1** | Other | **C级** | 普通客户 |

### 因子4：客户类型（个人客户识别）

| D365字段 | 值 | 标签 | 结果 |
|---------|-----|------|------|
| `mcs_accounttype` | **1** | Individual Account | **个人客户** |
| | **2** | Company Account | 公司客户 |

> ✅ **个人客户识别**: 使用 `mcs_accounttype=1`（Individual Account）识别个人客户，待业务方确认

## 代码实现

### CreditScorePlugin.cs 核心逻辑

```csharp
// 1. 查询Account客户属性
var account = service.Retrieve("account", accountId, 
    new ColumnSet("mcs_accountcategory", "mcs_accountlevel", "mcs_dealerrank"));

// 2. 判断是否经销商
int accountCategory = account.GetAttributeValue<OptionSetValue>("mcs_accountcategory")?.Value ?? 0;
bool isDealer = (accountCategory == 10 || accountCategory == 90);

// 3. 直销客户分级
int accountLevel = account.GetAttributeValue<OptionSetValue>("mcs_accountlevel")?.Value ?? 0;

// 4. 判断新老客户
bool isOldCustomer = CheckHasSalesOrder(service, accountId);

// 5. 匹配评分卡类型
return MatchScoringCardType(isOldCustomer, isDealer, accountLevel, accountCategory, tracer);
```

### MatchScoringCardType 方法

```csharp
private int MatchScoringCardType(bool isOldCustomer, bool isDealer, int accountLevel, int accountCategory, ITracingService tracer)
{
    // 个人客户判断（待业务确认）
    bool isPersonal = (accountCategory == 20 && accountLevel == 1);
    if (isPersonal) return 5;

    // 经销商
    if (isDealer)
    {
        return isOldCustomer ? 6 : 7;
    }

    // 直销客户：SA级 = S(4)或A(3)，BC级 = B(2)或C(1)
    bool isBigAccount = (accountLevel == 4 || accountLevel == 3);
    
    if (isOldCustomer)
    {
        return isBigAccount ? 1 : 3;
    }
    else
    {
        return isBigAccount ? 2 : 4;
    }
}
```

## 评分项目分档匹配规则

评分卡类型匹配完成后，系统会按评分项目（`mcs_credititem` / `mcs_itemid`）读取该类型下的所有评分卡分档配置，并依据以下规则计算单项得分。

### 配置规则

1. **同一评分项目可配置多行分档**  
   同一 `mcs_categoryid` + 同一评分项目下，支持配置多行评分卡记录，用于覆盖不同的取值区间或定性枚举值。

2. **定量区间语义**  
   定量分档区间统一采用 **左闭右开** `[mcs_minvalue, mcs_maxvalue)`：  
   - 最小值 `mcs_minvalue` **包含** 在本分档内；  
   - 最大值 `mcs_maxvalue` **不包含** 在本分档内；  
   - 因此配置时要求 `mcs_minvalue < mcs_maxvalue`。

3. **定性匹配值**  
   定性分档通过 `mcs_listvalue` 存储的原始枚举值（如 `L`、`M`、`H`）进行匹配，匹配时不区分大小写。

4. **赋分 `mcs_weight` 允许为负数**  
   为支持“扣分项”等业务场景，分档赋分可以是负数。代码层面不再对 `mcs_weight` 做非负校验。

### 匹配与算分规则

1. **每个标签只计算一次**  
   信用分计算时，一个客户标签记录只会命中同一评分项目下的一个分档，得分为对应分档的 `mcs_weight`。

2. **定量命中规则**  
   复核定量值（`mcs_itemintvalue2`）落在某一行 `[min, max)` 区间内时，即命中该行。若有多行区间满足条件，按数据库返回顺序取第一行。

3. **定性命中规则**  
   复核定性值（`mcs_itemtxtvalue2`）等于某一行 `mcs_listvalue`（忽略大小写）时命中该行。空字符串与 `O` 视为缺失值，不命中任何分档。

4. **缺失/未命中时取平均分**  
   当标签值为空、`N/A`、`O` 或没有任何分档命中时，该项目得分 = 该评分项目下所有分档赋分的 **整数平均分**（四舍五入，远离零方向）。  
   例如某项目配置三行分档权重为 `20、10、0`，则未命中时得分 = `(20+10+0)/3 = 10`。

5. **显式缺失值配置**  
   业务上如需“无/未提供资料”等缺失场景得固定分，建议为该评分项目单独配置一行对应枚举值或区间，权重设为期望分值（如 `0`），这样即避免走平均分逻辑。

### 配置校验

`CreditScoringCardValidationPlugin` 在 `mcs_credit_scoringcard` 的 Create/Update PreOperation 阶段执行校验：

- **定性值重复**：同一 `category + 评分项目` 下，不能存在两行配置相同的 `mcs_listvalue`。
- **定量区间重叠**：同一 `category + 评分项目` 下，任意两行的 `[min, max)` 区间不能重叠。
- **区间合法性**：`mcs_minvalue < mcs_maxvalue`。

重复或重叠时，保存会被阻断并返回错误信息。

## 从数据集成到审批反写的完整流程

> 本章节描述从触发"内外部数据集成"开始，到审批结果反写结束的完整取值和流转过程。表和字段均使用显示名称。

---

### 阶段1：触发数据集成（状态 10 → 11）

1. 用户在**客户信用评估记录表**表单上点击【下一步】。
2. 系统更新**客户信用评估记录表**的**状态**字段：从 `10 关联客户代码` 变为 `11 内外部数据集成`。
3. `CofaceDataSyncPlugin` 触发（Update PostOperation，监听**状态**字段变更）。

**取值来源**：
- **客户信用评估记录表**的**客户字段**（`mcs_accountid`）→ 关联到**客户表**
- **客户信用评估记录表**的**国家代码**（`mcs_countrycode`）
- **客户表**的**科法斯客户代码**（`mcs_cofaceid`），实际从**客户主数据表**的**科法斯客户代码**带出

---

### 阶段2：调用 Coface 接口获取外部数据

1. `CofaceDataSyncPlugin` 读取**系统配置表**中的 `CofaceApiConfig`，获取接口地址、认证信息。
2. 根据**国家代码**查询**系统配置表**中的 `CofaceCountryConfig`，确定调用的 Report 产品类型：
   - 非受限国家 → `customized-report` + `301`
   - 受限国家（39国，除 RU）→ `full-report`
   - RU → `customized-report` + `21000`
3. 调用 Coface 接口：
   - `URBA360` 监控接口（所有国家）
   - `Full Report` 内容接口（按国家类型选择产品）
4. 返回的 JSON 数据交给 `Urba360Parser` 和 `FullReportParser` 解析。

### Coface 财务指标配置表字段说明

财务指标（净资产、资产负债率、流动比率、净利润率）的科目编码按国家不同，通过 **Coface 财务指标国家配置表**（`mcs_coface_financial_indicator`）配置。

| 字段 | 显示名称 | 用途 |
|---|---|---|
| `mcs_countrycode` | 国家代码 | 按国家加载配置。不同国家的财务报表科目编码不同，解析时先按国家代码筛选。 |
| `mcs_indicatorname` | 指标名称 | 对应 Parser 中的指标名，如 `NetAssets`、`DebtRatio`、`CurrentRatio`、`NetProfitMargin`。 |
| `mcs_typevalue` | Coface类型编码 | 要匹配的 Coface JSON 中 `indicator.type.value` 或 `ratio.type.value` 的值。 |
| `mcs_priority` | 匹配优先级 | 同一指标可配置多条备选科目编码，按优先级从小到大依次尝试匹配，命中即停止。 |
| `mcs_indicatortype` | 取数位置 | 当前代码中仅读取、未实际使用。理论上区分从资产负债表还是比率取数，但目前 `NetAssets` 固定从资产负债表取，`DebtRatio/CurrentRatio/NetProfitMargin` 固定从比率取。 |
| `mcs_formulafallback` | 兜底计算公式 | 当前代码中仅读取、未实际使用。预留字段，用于直接匹配不到科目时通过公式计算指标值。 |

**实际取值流程（以 AR 国家净资产为例）**：
1. 解析 `NetAssets` 时，读取国家代码 = `AR`、指标名称 = `NetAssets` 的配置。
2. 按匹配优先级排序，先尝试 `type.value = 21`。
3. 在 URBA360 JSON 的 `productDetails.financials.balanceSheet.balanceSheetItems[].indicators[]` 中查找 `type.value = 21`。
4. 命中后读取 `fromAmount`，经单位换算（dimension）和货币转换（统一转 USD）后返回。
5. 若未命中，继续尝试同一指标的下一条优先级配置。

   | E | 高风险 |
   | O | 缺失 |

   **行业风险（SectorRisk）映射**

   | Coface 原始值 | 中文显示名称 |
   |--------------|-------------|
   | 1 | 低风险 |
   | 2 | 中风险 |
   | 3 | 高风险 |
   | 4 | 高风险 |
   | O | 缺失 |

   **外部评级（ExternalRating）映射**

   | Coface 原始值 | 中文显示名称 |
   |--------------|-------------|
   | 0 | 违约/资不抵债 |
   | 1 | 极端风险 |
   | 2 | 极高风险 |
   | 3 | 高风险 |
   | 4 | 显著风险 |
   | 5 | 中等风险 |
   | 6 | 可接受风险 |
   | 7 | 一般风险 |
   | 8 | 低风险 |

4. 最终在**客户信用标签表**生成若干条记录，每条关联到：
   - **客户信用评估记录表**（通过**评估记录**字段）
   - **客户信用评分项目表**（通过**评分项目**字段）
5. 数据集成完成后，**客户信用评估记录表**的**API状态**更新为 `SUCCESS`，**状态**自动推进为 `12 人工复核`。

---

### 阶段4：人工复核（状态 12 → 13）

1. 用户在**客户信用评估记录表**表单上查看生成的客户信用标签。
2. 对需要修正的指标，在标签上填写**复核定性值**或**复核定量值**。
3. 确认无误后点击【下一步】，**状态**从 `12 人工复核` 变为 `13 信用分计算`。

---

### 阶段5：信用分计算（状态 13）

1. `CreditScorePlugin` 触发（Update PostOperation，监听**状态**字段变更）。
2. 读取**客户表**的以下字段，匹配评分卡类型：
   - **客户类别**（`mcs_accountcategory`）
   - **客户级别**（`mcs_accountlevel`）
   - **客户类型**（`mcs_accounttype`）
3. 查询**销售订单表**，判断该客户是否有历史订单，确定新老客户。
4. 根据决策树得到**评分卡类型**（1~7），对应**客户评分卡配置表**的**评分卡类型**（`mcs_categoryid`）。
5. 遍历该评估记录下的所有**客户信用标签**：
   - 用标签的**评分项目编码**去匹配**客户评分卡配置表**中同类型、同项目的分档行
   - 定量标签：用**复核定量值**（优先）或**集成定量值**（`mcs_itemintvalue2` → `mcs_itemintvalue1`）匹配 `[min, max)` 区间
   - 定性标签：用**复核定性值**（优先）或**集成定性值**（`mcs_itemtxtvalue2` → `mcs_itemtxtvalue1`）匹配 `mcs_listvalue`
   - 未命中时，取该评分项目所有分档权重的平均分
6. 累加所有命中分档的**赋分**（`mcs_weight`），得到最终信用分。
7. 将信用分写入**客户信用评估记录表**的**信用分**字段（`mcs_creditscore`）。
8. 同时更新参与评分的标签：对成功命中分档并得到分数的**客户信用标签表**记录，将**是否评分**（`mcs_isscore`）字段更新为 `是`；未命中分档或值为缺失的标签保持为 `否`。

   > **是否评分字段赋值逻辑**
   >
   > | 时机 | 是否评分 | 说明 |
   > |---|---|---|
   > | 标签创建/初始化时 | 否 | `CofaceDataSyncPlugin` 和 `CustomerTagInitPlugin` 创建标签时默认值 |
   > | 信用分计算后 | 是 | `ScoreCalculator` 对命中分档并计算出分数的标签，回写得分时同步更新 |
   >
   > 画像页面、客户飞轮等后续展示时，通常只筛选 `是否评分 = 是` 的标签。

---

### 阶段6：审核申请（状态 13 → 14）

1. 用户点击【下一步】，**状态**从 `13 信用分计算` 变为 `14 审核申请`。
2. `BppIntegrationPlugin` 触发（Update PostOperation，监听**状态**字段变更）。
3. 调用 D365 自定义 API `mcs_bppstartapi`，传入：
   - **评估记录ID**
   - 实体名 `mcs_credit_record`
   - 当前用户ID
4. BPP 框架异步向 BPP 平台发起审批请求，**客户信用评估记录表**的**BPP状态**更新为 `Submitted`。
5. 审批状态同步后，**状态**推进为 `15 审批通过` 或 `16 审批未通过`。

---

### 阶段7：审批结果反写（状态 15）

1. `BppCallbackPlugin` 触发（Update PostOperation，监听**BPP状态**字段变更）。
2. 当**BPP状态**为 `Approved` 时：
   - 更新**客户信用评估记录表**的**状态**为 `15 审批通过`
   - 读取**客户信用评估记录表**的**信用分**、**客户等级**、**信用评估有效状态**
   - 通过**客户表**的**客户主数据**关联，回写到**客户主数据表**的以下字段：
     - **客户信用评分**（`mcs_creditscore`）
     - **客户等级**（`mcs_creditgrade`）
     - **信用评估有效状态**（`mcs_creditvalid`）
3. 当**BPP状态**为 `Rejected` 时，**状态**更新为 `16 审批未通过`，不回写客户主数据。
