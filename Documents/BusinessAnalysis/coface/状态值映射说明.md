# 评估状态字段值映射说明

> 创建日期: 2026-06-05
> 说明: 补充PRD中评估状态字段的选项集实际值

## 重要说明

PRD中提到的状态顺序（1-8）是**业务逻辑编号**，不是D365选项集的实际值。

D365选项集 `mcs_status` 的实际值如下：

| 业务顺序 | 状态名称 | 选项集实际值 | 说明 |
|---------|---------|------------|------|
| 1 | 发起信用评估 | **9** | 初始状态，新建记录时自动设置 |
| 2 | 关联客户代码 | **10** | 获取Coface ID阶段 |
| 3 | 内外部数据集成 | **11** | 调用Coface接口+内部数据 |
| 4 | 人工复核 | **12** | 补录/修订指标值 |
| 5 | 信用分计算 | **13** | 系统计算信用分 |
| 6 | 审核申请 | **14** | 提交BPP审批 |
| 7 | 审批通过 | **15** | BPP审批通过 |
| 8 | 审批未通过 | **16** | BPP审批驳回 |

### JS代码
```javascript
// 设置初始状态
statusField.setValue(9); // 发起信用评估

// 状态判断

    case 10: // 关联客户代码

    case 12: // 人工复核
    case 13: // 信用分计算
    case 14: // 审核申请
    case 15: // 审批通过

```

```csharp
// 触发条件

```

## BPF阶段配置

BPF（Business Process Flow）"信用评估记录流程"已配置8个阶段，每个阶段绑定 `mcs_status` 字段：

| BPF阶段 | 对应状态值 | 说明 |
|---------|-----------|------|
| 发起信用评估 | 9 | 初始阶段 |
| 关联客户代码 | 10 | 选择客户后自动推进 |
| 数据集成 | 11 | Coface数据+内部数据 |
| 人工复核 | 12 | 补录/修订指标值 |
| 信用分计算 | 13 | Plugin自动触发计算 |
| 审核申请 | 14 | 提交BPP审批 |
| 审批通过 | 15 | BPP审批通过 |
| 审批未通过 | 16 | BPP审批驳回 |

**注意：** BPF阶段切换由 `mcs_status` 字段值驱动，状态更新后BPF自动跳转到对应阶段。

## BPF阶段与状态值同步方案

### 问题描述
BPF前端显示的阶段和评估状态字段(mcs_status)没有自动对应。例如：
- BPF显示在"发起信用评估"阶段
- 但mcs_status字段值为13(信用分计算)

### 根因分析
D365 BPF的阶段推进默认通过用户点击"下一步"按钮或API更新`activestageid`字段实现。
BPF的ConditionJson配置（阶段条件）在D365中保存时会被清理/验证，无法通过后端代码直接配置。

### 解决方案
**采用Plugin同步方案**：当mcs_status更新时，Plugin自动查找并更新对应BPF实例的activestageid。

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    状态变更触发源                              │
├─────────────┬─────────────┬─────────────┬───────────────────┤
│  前端JS保存  │ 前端JS保存   │ 前端JS保存   │ CreditScorePlugin │
│  (9→10)     │ (10→11)     │ (11→12...)  │ (13→14)          │
└──────┬──────┴──────┬──────┴──────┬──────┴─────────┬─────────┘
       │             │             │                │
       └─────────────┴─────────────┴────────────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │ mcs_status 字段更新   │
              └──────────┬───────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │ BpfStageSyncPlugin   │  ← 统一监听mcs_status变更
              │ (Update, PostOp,     │     自动同步BPF activestageid
              │  filtering: mcs_status)
              └──────────┬───────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │ BPF实例activestageid  │
              │ 同步到对应阶段         │
              └──────────────────────┘
```

### 关键设计原则

**单一职责**：所有BPF同步逻辑集中在 `BpfStageSyncPlugin` 一个Plugin中，不分散到各个业务Plugin。

**原因**：
1. 如果每个业务Plugin（CreditScorePlugin/BppIntegrationPlugin）都自己调用SyncBpfStage，会导致：
   - 代码重复
   - 前端直接修改状态时（如用户手动改状态9→10）BPF不会同步
   - 维护困难，新增状态时需要修改多个Plugin

2. BpfStageSyncPlugin监听 `mcs_status` 字段变更，无论状态由谁更新（JS/Plugin/Flow），BPF都会自动同步。

#### 2. BpfStageSyncPlugin.cs（通用同步Plugin）
- **触发**：Update后, mcs_credit_record, mcs_status字段变更
- **阶段**：Post-operation (40)
- **模式**：Synchronous
- **Rank**：2（在CreditScorePlugin之后执行）
- **过滤字段**：mcs_status
- **功能**：调用BpfSyncHelper.SyncBpfStage同步BPF阶段

#### 3. 业务Plugin（不处理BPF同步）
- **CreditScorePlugin.cs**：只负责计算信用分，更新mcs_status=14
  - **不调用** SyncBpfStage（由BpfStageSyncPlugin统一处理）
- **BppIntegrationPlugin.cs**：只负责BPP审批集成，更新mcs_status=15/16
  - **不调用** SyncBpfStage（由BpfStageSyncPlugin统一处理）

### 状态流转时BPF同步点

| 状态变更 | 触发方式 | BPF同步 | 说明 |
|---------|---------|---------|------|
| 9→10 | JS保存表单 | BpfStageSyncPlugin自动同步 | 选择客户后保存 |
| 10→11 | JS保存表单 | BpfStageSyncPlugin自动同步 | 数据集成操作 |
| 11→12 | JS保存表单 | BpfStageSyncPlugin自动同步 | 人工复核 |
| 12→13 | JS保存表单 | BpfStageSyncPlugin自动同步 | 提交计算 |
| 13→14 | CreditScorePlugin | BpfStageSyncPlugin自动同步 | 计算完成后更新状态 |
| 14→15 | BppIntegrationPlugin | BpfStageSyncPlugin自动同步 | 审批通过后更新状态 |
| 14→16 | BppIntegrationPlugin | BpfStageSyncPlugin自动同步 | 审批驳回后更新状态 |
