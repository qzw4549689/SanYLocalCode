# Coface 汇率方案说明（FAQ）

> **文档用途**：解答客户关于“为何不使用 D365 自带汇率”的疑问  
> **适用对象**：业务人员、客户、项目管理人员  
> **最后更新**：2026-06-19

---

## 一、背景说明

在三一 D365 海外客户评分卡项目中，Coface 接口返回的财务数据（如注册资本、净资产等）默认使用**当地货币（Local Currency, LC）**。为了统一评分口径，项目要求将这些金额统一转换为 **USD（美元）**。

项目最终采用独立的 **`mcs_coface_exchange_rate`** 配置表来维护汇率，而不是使用 D365 系统自带的 `TransactionCurrency` / `Currency` 汇率。

---

## 二、常见问题

### Q1：为什么不使用 D365 自带的汇率？

D365 自带汇率主要服务于 CRM 内部交易货币字段（如报价单、订单、发票等），其设计目标与 Coface 数据集成的业务场景存在明显差异。直接使用会带来业务规则冲突、数据精度、系统影响面等多方面的问题。

---

### Q2：使用 D365 自带汇率会有哪些具体问题？

| 问题分类 | 具体说明 |
|---------|---------|
| **业务规则冲突** | 项目文档明确要求使用 **2026 Budget 年度固定汇率**，不建议实时汇率；D365 自带汇率通常是当前有效汇率，没有“年度预算”概念。 |
| **换算方向不一致** | D365 汇率通常基于组织基础货币（Base Currency，如 CNY），而项目要求统一转为 USD。若 Base Currency 不是 USD，需要二次换算，增加误差。 |
| **货币覆盖不足** | Coface 接口涉及 200 多种 ISO 货币代码，D365 通常只维护日常交易货币，大量小众币种未配置。 |
| **精度风险** | 越南盾（VND）、印尼盾（IDR）等小币种汇率值极小（如 0.00004），D365 标准汇率字段精度有限，容易产生换算误差。 |
| **缺少审计字段** | D365 标准汇率缺少生效日期、启用状态、来源备注等字段，无法满足按年度切换和审计追溯需求。 |
| **影响范围过大** | D365 汇率是全局配置，修改会影响报价、订单、发票等所有财务模块，风险不可控。 |
| **与现有实现冲突** | 当前代码、测试用例、UAT 验证均基于独立汇率配置表实现，切换需要大量返工。 |

---

### Q3：当前方案是如何设计的？

项目采用独立的 D365 自定义实体：**`mcs_coface_exchange_rate`**（Coface 汇率配置表），字段设计如下：

| 字段 | 说明 |
|------|------|
| `mcs_currencycode` | ISO 货币代码，如 EUR、CNY、USD |
| `mcs_currencyname` | 货币名称 |
| `mcs_rate_to_usd` | 1 单位本地货币可兑换的 USD 金额 |
| `mcs_effectivedate` | 生效日期，支持按年度维护多套汇率 |
| `mcs_isactive` | 是否启用，仅启用记录参与换算 |
| `mcs_remark` | 汇率来源或更新说明 |

---

### Q4：当前方案的优势是什么？

| 优势 | 说明 |
|------|------|
| **符合业务要求** | 与项目文档中“采用 2026 Budget 年度固定汇率、统一转 USD”的要求完全一致。 |
| **与 D365 核心配置隔离** | 不影响报价、订单、发票等其他业务模块的汇率设置。 |
| **灵活可维护** | 每年只需更新一张配置表即可切换新年度预算汇率，无需修改代码。 |
| **支持审计追溯** | 通过生效日期、启用状态、备注字段，可追溯每套汇率的来源和有效期。 |
| **容错友好** | 某货币未配置汇率时，系统保持原值并记录 Trace 日志，不会中断数据集成。 |
| **精度可控** | 独立字段可设置合适的 decimal 精度，避免小币种换算失真。 |

---

### Q5：2026 Budget 汇率表包含哪些货币？

当前项目主要覆盖 16 种常见货币，示例如下：

| 货币 | 代码 | 1 LC → USD |
|------|------|-----------|

| US Dollar | USD | 1.00 |
| Chinese Yuan | CNY | 0.1463 |
| Japanese Yen | JPY | 0.00725 |

| Indian Rupee | INR | 0.0122 |
| Polish Zloty | PLN | 0.26 |
| Vietnamese Dong | VND | 0.0000402 |

### Q6：如果未来客户坚持使用 D365 自带汇率，是否可行？

技术上可以调整，但需要评估以下成本和风险：

1. **需求变更确认**：需要客户正式确认放弃“年度预算汇率”方案，改用 D365 实时/交易汇率。
2. **代码改造**：需重写 `CofaceExchangeRateHelper`，并调整 `Urba360Parser`、`FullReportParser` 的换算逻辑。
3. **测试回归**：需重新执行 TC-CFPI-016、TC-CFPI-017 等汇率相关测试用例。
4. **UAT 重新验证**：已完成的 UAT 验证需要重新来过。
5. **系统影响评估**：需确认 D365 Base Currency、已有交易数据、报表等是否会受到影响。

建议在充分评估后，以正式变更请求（Change Request）方式处理。

## 四、结论

基于项目文档要求、数据准确性、系统影响面和维护成本综合考虑，**Coface 数据集成采用独立的 `mcs_coface_exchange_rate` 配置表维护 2026 Budget 年度固定汇率，而不是使用 D365 自带汇率**。

该方案既满足了客户原始需求，又与 D365 核心财务配置保持隔离，是当前项目背景下更稳妥、可维护性更高的选择。
