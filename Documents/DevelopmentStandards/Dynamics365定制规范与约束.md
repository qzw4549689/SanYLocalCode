## 一、核心规范

### 遵循微软"不破坏原生架构"的底层原则

所有定制必须基于 D365 官方开放的扩展点，禁止修改/依赖原生私有组件。

| 定制类型 | 核心规范 |
|---------|---------|
| **Dataverse 表扩展** | 仅扩展**自定义字段/表**，不修改系统表的核心字段（如原生主键、系统字段）；自定义表需遵循命名规范（前缀+语义，如 `new_marketingroi`）。 |
| **表单/视图定制** | 不覆盖原生表单的系统控件，仅通过"追加"方式新增自定义控件；视图筛选仅依赖公开字段，不依赖原生内部视图 ID。 |
| **代码扩展（插件/JS）** | 仅调用微软**公开的 API/SDK**，不依赖未文档化的内部方法、私有端点；插件仅注册到公开的事件管道（如 Create/Update）。 |
| **集成定制** | 仅使用官方开放的集成方式（如 Dataverse Web API、Power Automate 连接器），不直接访问 D365 服务器文件系统/数据库。 |

## 二、关键约束（红线）

以下行为会直接导致升级后定制内容不可用，**必须严格禁止**：

### 1. 依赖原生私有/未文档化的内容

| 禁止项 | 说明 |
|--------|------|
| **禁止依赖内部 API/端点** | 如调用以 `Microsoft.Dynamics.CRM.Internal` 开头的未公开 API、直接访问 Dataverse 的底层数据库表（如 `MSCRM_*` 物理表）；微软升级时会无通知修改这些内容。 |
| **禁止硬编码原生 ID** | 如硬编码原生表 ID、字段 ID、视图 ID、表单 ID（例：`83025000-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）；升级可能变更这些系统 ID，导致脚本/插件找不到目标对象。 |
| **禁止依赖原生 UI 结构** | 如通过 JavaScript 直接操作原生 DOM 元素（如 `document.getElementById("msdyn_marketingemail_name")`）；升级后原生 UI 的 DOM 结构会重构，脚本直接失效。 |

### 2. 修改原生系统组件

| 禁止项 | 说明 |
|--------|------|
| **禁止修改原生系统表/字段** | 如删除/重命名原生系统字段（如 `contact.emailaddress1`）、修改原生业务流程的核心步骤；升级时微软会重置被修改的系统组件。 |
| **禁止覆盖原生 Web 资源** | 如替换 D365 原生的 JavaScript 文件、CSS 样式；升级会强制还原原生 Web 资源，覆盖自定义修改。 |
| **禁止修改原生安全角色** | 如直接编辑"系统管理员""营销专业人员"等原生角色的权限；升级可能重置角色配置，导致权限失效。 |

### 3. 代码层面的高危操作

| 禁止项 | 说明 |
|--------|------|
| **禁止插件/脚本阻塞系统事件** | 如在原生核心事件（如 `contact.Create`）中编写长耗时同步插件、无限循环脚本；升级后微软会强化事件超时机制，直接终止这类代码。 |
| **禁止绕过 Dataverse 安全模型** | 如通过插件模拟系统管理员权限、直接修改权限元数据；升级会修复这类"漏洞"，导致代码执行失败。 |

## 三、防升级失效的核心实践

### 1. 定制设计阶段："松耦合"原则

- **使用逻辑名称而非 ID**：所有代码/配置中，均使用字段/表的**逻辑名称**（如 `new_marketingchannel`）而非 ID；逻辑名称由微软保证稳定，升级不会变更。

  ✅ **正确示例（JS）**：
  ```javascript
  formContext.getAttribute("mobilephone").getValue()
  ```
  
  ❌ **错误示例**：
  ```javascript
  formContext.getAttribute("83025000-xxxx").getValue()
  ```

- **避免直接依赖原生组件**：如需扩展原生功能，通过"封装层"隔离（如自定义操作封装原生 API 调用），升级时仅需修改封装层，无需重构所有代码。

- **自定义组件独立封装**：所有自定义字段、表、插件、Web 资源均添加统一前缀（如企业缩写 `cmp_`），与原生组件明确区分，避免升级时命名冲突。

### 2. 低代码配置阶段："扩展而非修改"

#### 表单/视图定制

- 原生表单仅做"追加式修改"（新增标签、字段、选项卡），不删除/隐藏原生核心字段；如需简化表单，创建**自定义表单**（而非修改原生主表单）。
- 视图筛选使用"字段逻辑名称+运算符"，不依赖原生视图的"保存查询 ID"；自定义视图单独保存，不覆盖原生系统视图。

#### 客户旅程/营销组件定制

- 自定义客户旅程步骤时，仅使用官方提供的"触发器/操作"（如"发送电子邮件""更新记录"），不自定义未公开的触发条件。
- 营销电子邮件模板使用官方支持的动态内容标签（如 `{{contact.fullname}}`），不使用自定义 HTML/JS 嵌入原生未支持的标签。

#### 解决方案管理

- 所有定制内容封装到**非托管解决方案**，且按"功能模块"拆分（如"营销字段扩展""营销表单脚本"），避免单一解决方案包含所有定制（升级后便于定位问题）。
- **禁止直接在生产环境修改定制内容**，所有变更需通过"开发→测试→生产"的解决方案部署流程。

### 3. 代码开发阶段：兼容与容错

#### 插件开发规范

- 使用最新版的 Dataverse SDK（如 `Microsoft.PowerPlatform.Dataverse.Client`），避免使用过时的 SDK 版本（如 2018 版）；微软会逐步淘汰旧 SDK 接口。
- 插件注册时选择"异步执行"（非核心场景），避免同步插件因升级后事件流程变更导致阻塞；同步插件需添加异常捕获（try-catch），并记录详细日志。
- 插件中使用 `IOrganizationService` 接口而非直接访问数据库，且调用 API 时指定"API 版本"（如 `v9.2`），避免默认使用过时版本：

  ```csharp
  // 正确：指定 API 版本
  var request = new RetrieveMultipleRequest
  {
      Query = new QueryExpression("contact") { TopCount = 10 },
      Parameters = new ParameterCollection { { "ApiVersion", "9.2" } }
  };
  ```

#### JavaScript 客户端开发

- 仅使用官方 `formContext` API（而非 `Xrm.Page`，已弃用），且添加字段/控件存在性校验：

  ```javascript
  function validateField(executionContext) {
      var formContext = executionContext.getFormContext();
      // 先校验字段是否存在，避免升级后字段被移除导致报错
      if (formContext.getAttribute("new_marketingchannel")) {
          var channel = formContext.getAttribute("new_marketingchannel").getValue();
          // 业务逻辑
      }
  }
  ```

- 避免直接操作 DOM，如需定制 UI，使用 Power Apps Component Framework（PCF）开发自定义组件（微软保证 PCF 组件的升级兼容性）。

#### API 集成规范

- 所有 Dataverse Web API 调用均指定**显式版本号**（如 `https://<org>.crm.dynamics.com/api/data/v9.2/`），不使用无版本的端点（如 `v9/`）；微软会维护最新版本的兼容性。
- 集成外部系统时，使用"松耦合"方式（如 Power Automate 连接器），避免硬编码外部 API 地址/密钥，通过 Dataverse 环境变量存储配置（升级时仅需修改环境变量）。

### 4. 升级前/升级后的验证流程

#### 升级前准备

1. **订阅微软 D365 更新公告**（[Dynamics 365 发布计划](https://docs.microsoft.com/dynamics365/release-plan)），提前了解升级内容（如哪些 API 将弃用、哪些组件将变更）。
2. 在**沙箱环境**中提前升级，完整测试所有定制内容：
   - 低代码配置：检查表单/视图/客户旅程是否正常加载、字段是否可编辑；
   - 代码扩展：触发插件/脚本/API，验证业务逻辑是否正常执行；
   - 集成场景：测试外部系统与 D365 的交互是否正常。
3. 对高风险定制（如依赖原生事件的插件），提前编写回滚方案（如临时禁用插件、恢复旧版解决方案）。

#### 升级后验证

1. 优先验证核心业务场景（如营销表单提交、客户旅程触发、数据同步）；
2. 检查自定义字段/表的权限是否正常（升级可能重置权限配置）；
3. 查看应用日志（Dataverse 插件日志、Power Automate 运行日志），定位报错内容；
4. 对失效的定制内容，优先通过"适配升级后的 API/配置"修复，而非修改原生组件。

### 5. 长期维护：持续兼容

- **定期清理冗余定制**：删除未使用的自定义字段、插件、Web 资源，减少升级时的兼容性风险；
- **跟进 API 弃用通知**：微软会提前 12 个月公布 API 弃用计划，需及时将旧 API 替换为新版（如将 `Xrm.Page` 替换为 `formContext`）；
- **使用微软认证的扩展方式**：如营销应用的定制优先使用官方提供的"营销扩展框架"，而非自定义开发的替代方案；
- **建立升级兼容性清单**：记录每个定制项的依赖项（如依赖的原生 API、组件），升级时快速核对是否受影响。

## 四、典型反例与修正方案

| 反例（升级必失效） | 修正方案（兼容升级） |
|-------------------|---------------------|
| 硬编码原生表单 ID 到 JS 脚本 | 使用表单逻辑名称 + `formContext` 获取表单，或通过"表单事件参数"动态获取 |
| 插件中直接访问 Dataverse 物理表 | 使用 `IOrganizationService` 接口 + 查询表达式（QueryExpression）访问数据 |
| 修改原生"营销电子邮件"表的核心字段 | 新增自定义字段扩展，而非修改原生字段；通过业务规则/插件关联自定义字段与原生字段 |
| JS 脚本直接操作原生 DOM 元素 | 使用 PCF 组件封装 UI 交互，或通过 `formContext` 操作控件属性（如 `setVisible`） |
| 未指定 API 版本的 Web API 调用 | 显式指定 API 版本（如 `v9.2`），并定期升级到最新稳定版本 |

## 五、总结

避免 D365 升级导致定制失效的核心逻辑是：**"不依赖原生私有内容、不修改原生系统组件、通过官方扩展点定制、提前验证适配"**。

具体落地时，需将"兼容性"纳入定制开发的全流程——设计阶段遵循松耦合原则，开发阶段使用官方稳定的 API/工具，升级阶段通过沙箱验证提前暴露问题，长期维护阶段跟进微软更新并持续适配。

**核心底线**：所有定制必须"可替换、可隔离、可验证"——即使升级导致某部分定制失效，也能通过修改定制内容（而非原生架构）快速恢复，且不影响其他功能。
