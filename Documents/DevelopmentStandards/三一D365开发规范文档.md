## 一、概述

D365开发分为3大部分：

| 部分 | 说明 |
|------|------|
| D365配置 | 实体、字段、窗体、视图、选项集、工作流等配置 |
| Azure服务开发 | MessageHandler、ExtensionApi、InnerApi等 |
| 移动端开发 | Android、iOS、H5前端开发 |

D365配置分3个环境：

| 环境 | 说明 |
|------|------|
| 开发环境（迭代环境） | 日常开发测试 |
| 测试环境（UAT环境） | 用户验收测试 |
| 生产环境 | 正式运行环境 |

开发迭代严格按照 **开发 → 测试 → 生产** 流程进行，**2周一个版本**。

## 三、开发规范

### 3.1 禁止项（红线）

以下行为 **严格禁止**，违反可能导致严重生产事故：

| # | 禁止项 | 详细说明 | 正确做法 |
|---|--------|---------|---------|
| 1 | **严禁直接推送强行覆盖** | 不得未经合并直接推送代码覆盖远端 | 先拉取UAT分支最新代码，本地合并到自己的开发分支处理完冲突后，再推送到远端，再合并到UAT/BugFix分支 |
| 2 | **严禁XrmToolBox连接非授权环境** | 禁止用XrmToolBox连接除迭代和预发布这两个环境外的其他环境 | 仅在迭代环境和预发布环境使用XrmToolBox注册、修改插件和CustomAPI |
| 3 | **严禁非授权环境直接修改实体** | 禁止在除迭代和预发布这两个环境外的其他环境上直接修改实体（包括字段、窗体、视图、选项）、Web资源等 | 所有实体修改在迭代环境完成，通过解决方案发布流转 |
| 4 | **严禁删除已存在的表、字段等资源** | 不得删除已经存在的表、字段等任何解决方案中的资源 | 如需要废弃插件，先将插件中内容注释，再在迭代或预发布分支禁用，再走正常发布流程 |
| 5 | **封板期禁止配置** | 每个版本发布生产期间为封板期 | 禁止在D365开发环境进行配置，防止将未测试配置发布到生产环境。待部署成功并导入Pre环境后进行下个版本的配置（一般持续1天，部署负责人会在飞书群通知） |
| 6 | **严禁插件监听所有字段** | 插件注册时必须明确指定监听的字段 | 注册插件时精确指定需要监听的字段，不得选择"所有字段" |
| 7 | **严禁直接覆盖配置项数据** | 更新 `ms_systemconfiguration` 等配置项数据前，必须先查询现有值，在原有数据基础上追加/合并后再更新，不得直接覆盖整记录 | 先 Retrieve 查询当前内容，解析后合并新配置，最后 Update 更新；更新后需校验未影响其他业务模块配置 |

### 3.2 注意事项

| # | 事项 | 说明 |
|---|------|------|
| 1 | **日志管理** | 往Application Insights写日志（messagehandler、innerApi、extensionApi）的日志，开发调试后 **一定记得注释掉**。日志容易写满，当天日志一旦写满之后就不会输出任何日志。特别是批量跑数据或者定时任务里面大量的日志以及异常日志 |
| 2 | **前后端校验分工** | 格式校验与简单业务规则放在前端，业务逻辑、数据校验、安全校验等 **必须放在后端** |
| 3 | **注册前拉取最新代码** | 注册customApi、plugin、webresource前都务必拉取最近代码再注册，**不要覆盖别人的功能**！应该保持自己本地代码是最新的 |

### 3.3 实体命名规范

- 解决方案新建的实体默认应该以 **`mcs_`** 开头
- 实体命名除 `mcs_` 外 **不要再加入下划线驼峰** 等
- 请参考现有实体：`mcs_serviceorder`

## 四、迭代流程

### 4.1 Git分支管理

```
main  ─────────────────────────────────────────────────
       |        |              |              |
uat    ─┬───────┬──────────────┬──────────────┬────────
        |       |              |              |
feature1 o─o─o──┘              |              |
feature2           o─o─────────┘              |
hotfix1                      o─o────────────────┘
feature3                                       o─o─o
```

#### 分支说明

| 分支 | 用途 |
|------|------|
| `main` | 生产分支，从main分支打包发布 |
| `uat` | 用户验收测试分支 |
| `feature` | 功能开发分支，可个人单独拉或按功能小团队共用 |
| `hotfix` | 紧急修复分支 |

#### 分支命名规范

| 命名方式 | 示例 | 适用场景 |
|---------|------|---------|
| 版本+个人域账号 | `0319-yangjh` | 个人开发分支 |
| 版本+功能命名 | `0319-repairmaterial` | 功能团队分支 |

### 4.2 代码开发流程

项目主要有 **MessageHandler、ExtensionApi、InnerApi**。

| 步骤 | 操作 | 说明 |
|------|------|------|
| a | 从远端UAT拉取开发分支 | 分支可以每个人单独拉也可以按功能等小团队共用一个 |
| b | 功能开发 | 按需求进行代码开发 |
| c | 本地单元测试 | 完成本地自测 |
| d | 拉取远端UAT/BugFix分支合并到当前开发分支并解决冲突 | 确保代码最新 |
| e | 推送到远端仓库 | 推送开发分支 |
| f | Web端创建Pull Request合并到UAT分支 | [Azure DevOps PR页面](https://dev.azure.com/SANYCRM/SANY%20Overseas%20CRM/_git/SANY%20Overseas%20CRM/pullrequests?_a=mine) |
| g | 迭代结束前测试通过由专人将代码从UAT分支合并到main分支 | 从main分支打包发布 |
| h | 回退功能记录 | 如果有回退的功能，在 **D365回退功能记录表** 中记录 |

### 4.3 D365配置流程

```
+------------------------+     +-------------------+     +-------------------+
|   解决方案开发完成      | --> |   迭代环境         | --> |   迭代测试通过     |
|   (四种解决方案类型)    |     |   (开发测试环境)    |     |                  |
+------------------------+     +-------------------+     +-------------------+
                                                                 |
+------------------------+     +-------------------+             |
|   生产环境部署          | <-- |   UAT环境         | <-----------+
|   (正式运行环境)        |     |   (用户验收测试)   |
+------------------------+     +-------------------+
                                      |
                               业务验收通过
                               所有解决方案同步
```

#### 特别注意

> 注册customApi、plugin、webresource前都务必拉取最近代码再注册，**不要覆盖别人的功能！！** 应该保持自己本地代码是最新的。

### 4.4 解决方案发布规范

#### 四种解决方案类型

| 解决方案 | 包类型 | 说明 |
|---------|--------|------|
| 实体元数据解决方案 | **非托管增量包** | entity_XX，XX代表版本号 |
| CustomAPI解决方案 | **托管全量包** | McsCustomAPI |
| Plugin解决方案 | **托管全量包** | McsPlugin |
| WebResource解决方案 | **托管全量包** | McsWebResource |

#### 每次迭代发布清单

| 解决方案 | 类型 | 内容 | 注意事项 |
|---------|------|------|---------|
| McsWebResource | 托管包 | js、html、翻译文件json等 | **添加时注意不要将实体、站点地图、OptionSet等其他资源带入** |
| McsOptionSet | 托管包 | 实体关联的下拉选 | |
| role_XX | 非托管包 | XX代表版本号，如role_20250319。每个版本一个包，修改的角色配置加入解决方案 | |
| McsCustomAPI | 托管包 | 自定义API | |
| McsPlugin | 托管包 | 自定义插件 | |
| McsAutomate | 托管包 | 自定义工作流 | |
| entity_XX | 非托管包 | XX代表版本号，如entity_20250319。实体、站点地图等。修改或添加实体、站点地图时将要修改的资源加入解决方案，或直接新建实体 | 实体命名以mcs_开头 |

#### 发布步骤

1. 在迭代环境配置相应的解决方案
2. 迭代环境自测
3. 发布到UAT环境测试
   - 发布时注意：**除实体包entity_XX与角色role_XX选择非托管包外，其他解决方案都选择托管包**

   - 登录账号：`caoy815@sany.com.cn` / `Sany318318`
4. 迭代结束前测试通过，由专人从迭代环境将解决方案发布到生产环境
