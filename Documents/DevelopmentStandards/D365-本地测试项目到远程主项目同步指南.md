# D365 本地测试项目 → 远程主项目同步指南

> 适用场景：本地在 `Code/Customizations/Plugins/` 下的独立测试项目（如 `CofaceIntegration`、`BppIntegration`）开发验证通过后，需要把代码归并到远程主项目 `SanyD365.D365Extension.Sales`，最终编译发布到 D365 DEV/UAT。
>
> 核心原则：**本地为源，远程为镜像**。除远程 `PluginBase` 框架脚手架外，业务逻辑必须与本地一个字符不差。

---

## 1. 项目结构差异

| 维度 | 本地测试项目 | 远程主项目 |
|---|---|---|
| 路径 | `Code/Customizations/Plugins/CofaceIntegration` | `D365/SanyD365.D365Extension.Sales` |
| 框架 | `IPlugin`（独立引用 CRM SDK） | `PluginBase`（项目内公共基类） |
| 目标 Framework | .NET Framework 4.6.2 | .NET Framework 4.6.2 |
| C# 版本 | 本地可用 C# 8+ | 远程主项目锁定 **C# 7.3** |
| 命名空间 | `SanyD365.Plugins.CofaceIntegration.*` | `SanyD365.D365Extension.Sales.*` |
| 编译产物 | 独立 DLL（测试用） | `SanyD365.D365Extension.Sales.dll`（发布用） |

## 5. 同步步骤

### 5.1 方式一：使用自动化脚本（推荐）

见 [`Code/Tools/sync-plugin-to-remote.py`](../../Code/Tools/sync-plugin-to-remote.py)。

```bash
cd Code/Tools
python3 sync-plugin-to-remote.py
```

脚本会：
1. 按映射表读取本地文件
2. 替换命名空间
3. 写入远程服务器对应目录
4. 自动更新 `SanyD365.D365Extension.Sales.csproj`
5. 触发远程单独编译验证

### 5.2 方式二：手动同步

1. 在本地复制需要同步的文件
2. 按命名空间映射表做文本替换
3. 按目录结构映射表放到远程对应位置
4. 打开 `SanyD365.D365Extension.Sales.csproj`，新增 `<Compile Include="..." />` 引用
5. 在远程服务器执行单独编译：
   ```powershell
   cd C:\Projects\D365\D365\SanyD365.D365Extension.Sales
   msbuild SanyD365.D365Extension.Sales.csproj /p:Configuration=Release /p:Platform=AnyCPU /verbosity:minimal
   ```

## 6. C# 7.3 常见降级点

本地开发时如果用了 C# 8+，同步到远程主项目必须降级。

### 6.1 switch 表达式 → switch 语句

```csharp
// ❌ C# 8 switch 表达式
int? value = creditGrade?.ToUpperInvariant() switch
{
    "A0" => 100000000,
    "A1" => 100000001,
    _ => null
};

// ✅ C# 7.3 switch 语句
int? value = null;
switch (creditGrade?.ToUpperInvariant())
{
    case "A0": value = 100000000; break;
    case "A1": value = 100000001; break;
}
```

### 6.2 using 声明 → using 语句

```csharp
// ❌ C# 8 using 声明
using var client = new HttpClient();

// ✅ C# 7.3 using 语句
using (var client = new HttpClient())
{
    // ...
}
```

### 6.3 空合并赋值 `??=` → 传统写法

```csharp
// ❌ C# 8
name ??= "default";

// ✅ C# 7.3
if (name == null) name = "default";
```

### 6.4 索引和范围 → 传统方法

```csharp
// ❌ C# 8
var slice = arr[1..3];

// ✅ C# 7.3
var slice = arr.Skip(1).Take(2).ToArray();
```

## 7. 编译验证

### 7.1 推荐：单独编译目标项目

```powershell
cd C:\Projects\D365\D365\SanyD365.D365Extension.Sales
msbuild SanyD365.D365Extension.Sales.csproj /p:Configuration=Release /p:Platform=AnyCPU /verbosity:minimal
```

> 注意：远程 `csproj` 默认 `Platform=AnyCPU`（无空格），单独编译时必须用 `AnyCPU`；Solution 编译时才用 `"Any CPU"`。

### 7.2 全量 Solution 编译（可选）

```powershell
cd C:\Projects\D365
nuget restore D365\D365.sln
msbuild D365\D365.sln /p:Configuration=Release /p:Platform="Any CPU" /verbosity:minimal
```

> ⚠️ 全量编译常因测试项目缺 `Secret.json` 失败，这是环境配置问题，不是代码问题。只验证目标项目即可。

## 8. Git / PR 流程

1. 在远程服务器创建个人分支：
   ```powershell
   cd C:\Projects\D365
   git checkout uat
   git pull origin uat
   git checkout -b uat-<日期>-<姓名>-<功能简述>
   ```

2. 提交修改：
   ```powershell
   git add .
   git commit -m "feat: xxx"
   git push -u origin uat-<日期>-<姓名>-<功能简述>
   ```

3. 在 Azure DevOps 创建 PR 合并到 `uat`
4. PR 合并后，在远程 `git checkout uat && git pull origin uat`
5. 重新编译目标项目，传回 DLL 更新 DEV Assembly

## 9. 发布到 DEV / UAT 流程

详见 `/skill:d365-deploy`。关键顺序：

1. **UAT 元数据发布**（如果涉及实体/字段/选项集变更）
2. **本地注册 DEV Assembly**（用 MetadataTool 更新 `SanyD365.D365Extension.Sales`）
3. **DEV 验证**
4. **用户手动通过 n8n Release Tool 发布到 UAT**

> AI 不得擅自执行 n8n UAT 发布，必须整理信息后由用户操作。

## 10. 经验教训

1. **不要全量编译 D365.sln 来验证单个项目**：测试项目依赖 `Secret.json`，会因环境文件缺失报错，浪费排查时间。
2. **本地为源，远程为镜像**：同步后要用 diff 工具对比本地和远程的业务逻辑是否一致。
3. **提前确认 C# 版本差异**：本地写代码时尽量用 C# 7.3 兼容语法，减少同步时改写成本。
4. **只同步自己修改的文件**：远程目录可能有其他人的代码，不要整目录覆盖。
5. **csproj 引用必须更新**：新增文件后务必加入 `<Compile Include>`，否则远程编译不报错但新代码不会生效。
6. **Assembly 差异红线**：更新 DEV Assembly 前必须先检查 PluginType 集合是否一致，不一致立即停止通知用户。
