# D365 + AI 敏捷实施方法论

**版本：** V2.0  
**日期：** 2026年6月  
**适用场景：** 基于 Microsoft Dynamics 365 的定制化开发项目，结合 AI 辅助开发

---

## 一、方法论概述

### 1.1 核心理念

> **AI 负责代码生成和重复性工作，人类负责业务确认和决策判断。**

本方法论总结了 D365 定制化开发项目中，利用 AI 辅助开发的完整实践经验，覆盖从需求确认到项目归档的全生命周期。

### 1.2 方法特点

| 特点 | 说明 |
|------|------|
| **文档驱动** | 所有开发基于 PRD 文档和数据字典，确保业务一致性 |
| **实体先行** | 先完成数据模型，再构建界面和逻辑 |
| **可视化验证** | 通过 SolutionViewer 实时查看实体结构，减少沟通成本 |
| **迭代微调** | 字段类型、选项值等细节通过对话快速调整 |
| **工具链闭环** | 从设计 → 开发 → 验证 → 导出，形成完整闭环 |
| **4核心文件体系** | 开发计划 + 测试用例 + Bug记录 + 日报，AI协作标准化 |
| **功能编号追踪** | F1.x~F6.x 贯穿所有文档，实现功能-测试-Bug 全链路追踪 |

### 1.3 多 AI 协作模式

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  AI 开发助手 │ ←→ │  AI 测试助手 │ ←→ │  AI 文档助手 │
│  (代码生成)  │     │  (测试执行)  │     │  (日报整理)  │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           ↓
                    ┌─────────────┐
                    │   人类用户   │
                    │ (业务确认/决策)│
                    └─────────────┘
```

## 二、4核心文件体系（项目基石）

任何 AI 接手项目，**必须先建立或确认这 4 个文件**。

| 文件 | 路径 | 用途 | 更新时机 | 负责 AI |
|------|------|------|----------|---------|
| **开发计划** | `Documents/Planning/开发计划.md` | 功能清单 + 进度跟踪 + 功能-测试-Bug 关联总表 | 每完成一个功能 | 开发 AI |
| **测试用例总集** | `Documents/Tests/测试用例总集.md` | 所有测试场景，TC-{MODULE}-{NNN} 编号 | 每新增/修改功能 | 测试 AI |
| **Bug 反馈记录** | `Documents/Tests/BugReports/测试Bug反馈记录.md` | Bug 追踪，含功能编号、修复记录、根因分析 | 每发现/修复 Bug | 测试 AI |
| **工作日报汇总** | `Documents/DailyReports/工作日报汇总.md` | 按日期章节的工作记录，表格化核心数据 | 每个工作日结束 | 文档 AI |

### 2.1 功能编号体系（F1.x ~ F6.x）

根据项目实际阶段划分，示例：

```
F1.x — Phase 1: 客户主数据扩展（实体/字段/视图/表单/SiteMap）
F2.x — Phase 2: 业务配置管理（配置项/规则/克隆导入导出）
F3.x — Phase 3: 业务流程管理（CRUD/状态机/外部数据集成/人工复核/工作流状态条UI）
F4.x — Phase 4: 核心计算引擎（匹配算法/逐项计算/特殊处理/等级映射/结果保存）
F5.x — Phase 5: 审批与外部集成（审批记录/虚拟审批/状态同步/第三方对接）
F6.x — Phase 6: 系统管理（批量处理/权限/日志/监控）
```

**使用规则：**
- 开发计划中每个功能项必须有唯一功能编号
- 测试用例必须标注「功能编号」列
- Bug 记录必须标注「功能编号」列
- 日报中涉及的功能必须引用功能编号

### Phase 1：实体建模（AI 辅助）

#### 3.3 目标

完成 D365 中所有实体的创建，包括实体本身和字段定义。

#### 3.4 工具链

```
PRD + 数据字典
    ↓
AI 生成实体定义 JSON
    ↓
C# CLI 工具创建实体
    ↓
D365 在线环境验证
    ↓
导出 Solution ZIP
    ↓
SolutionViewer 可视化确认
```

#### 3.5 详细步骤

**Step 1：实体定义 JSON 生成**

基于数据字典，AI 生成实体定义 JSON 文件：

```json
{
  "entityName": "new_assessment_record",
  "displayName": "评估记录表",
  "solutionName": "solution_20260603_dev",
  "fields": [
    {
      "schemaName": "new_recordcode",
      "displayName": "记录编码",
      "type": "string",
      "maxLength": 50,
      "required": true
    }
  ]
}
```

**要点：**
- 一个 JSON 对应一个实体
- 字段类型映射：Single Line of Text → string, Option Set → picklist, Lookup → lookup
- 中文显示名必须准确

**Step 2：实体创建（C# CLI）**

使用自研 `D365MetadataTool` 工具：

```bash
# 创建实体和字段
dotnet run create <实体定义.json>

# 检查解决方案中的实体
dotnet run check <解决方案名>

# 添加实体到解决方案
dotnet run add <实体名> <解决方案名>

# 发布实体
dotnet run publish <实体名>
```

**Step 3：字段类型调整**

初始创建时可能使用简单类型（如 string），后续根据业务需求调整为复杂类型：

| 调整场景 | 操作 | 命令 |
|---------|------|------|
| String → Lookup | 删除旧字段，新建 Lookup 字段 | `delete-field` + `create` |
| 补充选项集值 | 给已有 Picklist 字段插入选项 | `update-options` |
| 新增字段 | 直接创建 | `create` |

**注意：** 被引用的字段无法删除，需用新名字创建。

**Step 4：验证与导出**

```bash
# 列出实体所有字段
dotnet run list-fields <实体名>

# 导出解决方案 ZIP
dotnet run export <解决方案名> <输出路径>
```

**Step 5：可视化确认（SolutionViewer）**

启动 SolutionViewer Web 应用，查看：
- 实体列表
- 字段列表（自定义/系统分离）
- 选项集值
- Lookup 关联关系

```bash
cd Code/Tools/SolutionViewer
node server.js
# 访问 http://localhost:3456
```

#### 3.6 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 实体创建后不在解决方案中 | 未执行 add 命令 | `dotnet run add <实体> <解决方案>` |
| 字段类型错误 | 初始设计考虑不周 | 删除重建（如 String → Lookup） |
| 选项集值为空 | 中文显示名未翻译 | 回退到英文显示名 |
| 字段删除失败 | 被其他组件引用 | 使用新名字创建，保留旧字段 |
| 选项集无法更新 | 已有字段不能修改选项 | 使用 `InsertOptionValue` API |

### Phase 2：表单与逻辑开发

#### 3.7 目标

为每个实体配置 D365 表单，开发 JavaScript 表单逻辑和 C# Plugin 业务逻辑。

#### 3.8 开发内容分类

| 逻辑类型 | 示例 | 实现方式 | 负责层 |
|---------|------|---------|--------|
| **自动生成** | 编码规则（前缀+日期+序号） | Plugin (PreOp Create) | 后端 |
| **带入带出** | 选择客户带出关联信息 | JavaScript OnChange | 前端 |
| **显隐控制** | 根据类型显示不同字段组 | JavaScript OnLoad/OnChange | 前端 |
| **默认值** | 日期默认今天、申请人默认当前用户 | JavaScript OnLoad | 前端 |
| **校验规则** | 编码唯一性、必填项、数值范围 | JavaScript OnSave + Plugin PreOp | 前后端 |
| **状态流转** | BPF 按钮驱动状态变更 | Ribbon + Plugin PostOp | 后端 |
| **字段只读** | 编码字段、带出字段不可编辑 | JavaScript OnLoad | 前端 |
| **下拉联动** | 上级选项变更→过滤下级选项 | JavaScript OnChange + Web API | 前端 |

#### 3.9 开发顺序建议

```
1. 窗体字段摆放（C# CLI update-form）
2. 默认视图列配置（C# CLI update-view）
3. JavaScript 表单脚本开发
   3.1 编码字段只读
   3.2 带出字段只读
   3.3 自动带出逻辑（OnChange）
   3.4 显隐控制
   3.5 下拉联动
   3.6 数值校验
   3.7 保存前校验（OnSave）
4. Plugin 开发
   4.1 AutoNumber Plugin（编码生成）
   4.2 Validation Plugin（字段校验）
   4.3 Business Logic Plugin（核心计算等）
5. JS 部署与表单绑定
6. Plugin 注册与 Step 配置
7. 实体发布
```

#### 3.10 JS 开发规范

```javascript
// 1. 命名空间隔离
var AssessmentForm = AssessmentForm || {};

// 2. 严格校验实体名（防止误触发）
AssessmentForm.onLoad = function(executionContext) {
    var formContext = executionContext.getFormContext();
    // 逻辑...
};

// 3. 使用逻辑名而非硬编码 ID
formContext.getAttribute("new_accountid").getValue();  // ✅
formContext.getAttribute("83025000-xxxx").getValue();   // ❌
```

#### 3.11 Plugin 开发规范

```csharp
// 1. 开头严格校验实体名
if (context.PrimaryEntityName != "new_assessment_record")
    return;

// 2. 获取目标实体
var target = (Entity)context.InputParameters["Target"];

// 3. Sandbox 隔离模式
// 4. 仅注册到公开事件管道（Create/Update/Delete）
// 5. 不使用未文档化的内部 API
```

#### 3.12 部署流程

| 组件 | 部署命令 | 验证方式 |
|------|---------|---------|
| JS WebResource | `dotnet run deploy-js <文件> <实体>` | D365 界面表单加载检查 |
| JS 绑定表单 | 修改 systemform.formxml | 事件查看器确认绑定 |
| Plugin Assembly | `dotnet run register-plugin <dll>` | Plugin 注册工具查看 |
| Plugin Step | 代码注册或手动创建 | 测试触发验证 |
| 实体发布 | `dotnet run publish <实体>` | SolutionViewer 确认 |

#### 3.13 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Script Error：方法不存在 | 复制窗体时未改 JS 方法名 | 同步修改 `functionName` + JS 对象名 |
| 字段插入到 footer | `LastIndexOf("</rows>")` 匹配 footer | 在第一个 `<control>` 后的 `</row>` 处插入 |
| `cell id` 无效 GUID | 使用了非 GUID 格式 | `Guid.NewGuid().ToString("B")` |
| Plugin 误触发其他实体 | 未校验 `LogicalName` | 代码开头严格校验 |
| JS 绑定不生效 | 窗体 XML 未正确更新 | 检查 `formxml` 中 `event` 节点 |

### Phase 3：视图与 BPF 配置

#### 3.14 目标

配置列表视图列、BPF 业务流程阶段、BPF 阶段同步机制。

#### 3.15 视图配置

与表单更新类似，通过 Dataverse API 直接修改 `savedquery` 实体：

```csharp
// 查询默认 Public 视图 (querytype=0, isdefault=true)
var query = new QueryExpression("savedquery")
{
    Criteria = {
        Conditions = {
            new ConditionExpression("returnedtypecode", ConditionOperator.Equal, entityName),
            new ConditionExpression("querytype", ConditionOperator.Equal, 0),
            new ConditionExpression("isdefault", ConditionOperator.Equal, true)
        }
    }
};
```

视图列格式：`<cell name="new_fieldname" width="150" />`

#### 3.16 BPF 阶段配置

**操作步骤：**
1. 停用 BPF（Draft 状态才能修改 XAML）
2. 获取当前 XAML，解析现有阶段结构
3. 生成新阶段的 XAML 片段（每个阶段绑定状态字段）
4. 更新阶段链（NextStageId 指向下一阶段）
5. 插入新阶段到 XAML 中
6. 更新 workflow 记录
7. 创建 processstage 记录
8. 重新激活 BPF

**验证项：**
- BPF 状态：已激活（State=1）
- 阶段总数正确
- 阶段链完整无断点

#### 3.17 BPF 阶段同步 Plugin

**问题：** BPF 前端显示的阶段与状态字段值不对应。

**解决方案：** 当状态字段更新时，Plugin 自动查找并更新 BPF 实例的 `activestageid`。

```csharp
// 状态值 → StageId 映射（示例）
var stageMap = new Dictionary<int, Guid> {
    { 1,  new Guid("xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx") }, // 阶段1
    { 2,  new Guid("xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx") }, // 阶段2
    // ...
};
```

**注册要点：**
- 触发时机：Update of 主实体, PostOp
- Rank：2（在业务 Plugin 之后执行）
- Filter：仅当状态字段变更时触发

#### 3.18 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| BPF 阶段与状态值不对应 | ConditionJson 无法通过后端修改 | 使用 Plugin 同步方案 |
| D365 未自动解析新 Plugin Type | 解析缓存或延迟 | 手动创建 plugintype 记录 |
| BPF XAML 更新失败 | 未停用 BPF 就修改 | 必须先停用（Draft 状态） |

#### 3.19 目标

#### 3.20 接口分类

| 类型 | 示例系统 | 接口 | 状态 |

#### 3.23 常见问题与解决

### Phase 5：测试与质量保障

#### 3.24 目标

建立完整的测试体系，确保功能正确性和稳定性。

#### 3.25 测试策略

| 测试类型 | 执行方 | 工具 | 覆盖范围 |
|---------|--------|------|---------|

| **UAT** | 人类用户 | 手动 | 业务场景端到端 |

#### 3.27 Bug 反馈规范

**Bug 记录必须包含：**

| 字段 | 说明 |
|------|------|

#### 3.28 测试执行流程

```
1. 开发 AI 完成功能开发
   ↓
2. 更新开发计划状态 → 补充测试用例
   ↓
3. 测试 AI 执行测试用例
   ↓

   未通过？→ 记录 Bug → 开发 AI 修复 → 回归测试
   ↓

   ↓
```

#### 3.29 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 测试用例与功能不匹配 | 功能变更后未同步更新用例 | 功能变更时必须同步更新关联用例 |
| Bug 重复记录 | 未先搜索已有 Bug | 记录前先搜索现有 Bug 列表 |
| 回归测试遗漏 | 修复后未执行相关用例 | 建立回归测试清单 |

## 四、AI 协作模式

### 4.1 AI 适用场景

| 场景 | AI 作用 | 人类作用 |
|------|---------|---------|
| 代码生成 | 生成 C# / JavaScript / JSON | 审查逻辑正确性 |
| 文档整理 | 生成日报、开发计划、测试用例 | 确认内容准确性 |
| 数据分析 | 解析 ZIP/XML 结构、接口返回 | 解释业务含义 |
| 重复操作 | 批量字段创建、视图更新 | 监控执行结果 |
| 测试执行 | 执行 Playwright E2E、界面操作 | 确认业务场景覆盖 |
| Bug 分析 | 根因分析、修复方案建议 | 确认修复方案 |

### 4.2 多 AI 协作流程

```
┌─────────────────────────────────────────────────────────────┐
│                        人类用户                               │
│              （业务确认 / 决策判断 / 最终验收）                  │
└─────────────────────────────────────────────────────────────┘
                              ↑↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  AI 开发助手 │←→→│  AI 测试助手 │←→→│  AI 文档助手 │
│             │    │             │    │             │
│ • 实体建模   │    │ • 用例设计   │    │ • 日报整理   │
│ • JS/Plugin │    │ • 测试执行   │    │ • 文档归档   │
│ • Bug 修复  │    │ • Bug 记录   │    │ • 计划更新   │
│ • 接口开发  │    │ • 回归测试   │    │ • 知识沉淀   │
└─────────────┘    └─────────────┘    └─────────────┘
       ↑↓                 ↑↓                 ↑↓
   开发计划.md        测试用例总集.md      工作日报汇总.md
   （功能编号）        （TC-编号）          （日期章节）
       ↑↓                 ↑↓                 ↑↓
       └─────────────────┼──────────────────┘
                         ↓
                  测试Bug反馈记录.md
                    （Bug-编号）
```

### 4.3 关键原则

1. **业务规则必须人工确认** - AI 不替代业务判断
2. **数据字典是最高权威** - 所有开发以数据字典为准
3. **功能编号全链路追踪** - 开发→测试→Bug→日报，全部引用功能编号
4. **小步快跑** - 单个字段/实体调整，立即验证
5. **可视化验证** - 通过 SolutionViewer 减少沟通成本
6. **文档即代码** - 4核心文件与代码同等重要，必须同步更新

## 五、已知陷阱库（血泪教训）

### 5.1 窗体与 JS 陷阱

| 陷阱 | 后果 | 解决方案 |
|------|------|---------|
| **复制窗体时忘记改 JS 方法名** | Script Error，表单无法加载 | 同步修改 `functionName` + JS 对象名 + 文件名 |
| **`LastIndexOf("</rows>")` 匹配 footer** | 字段插入到 footer，Power Apps 不显示 | 在第一个 `<control>` 后的 `</row>` 处插入 |
| **`cell id` 不是有效 GUID** | FormXml schema 验证失败 | 使用 `Guid.NewGuid().ToString("B")` |
| **硬编码不存在的字段名** | 实体未找到 / 编译错误 | 先 `list-fields` 查询实际字段 |

### 5.2 Plugin 陷阱

| 陷阱 | 后果 | 解决方案 |
|------|------|---------|
| **Plugin 未校验实体名** | 误触发其他实体 | 代码开头严格 `if (target.LogicalName != "xxx") return;` |
| **Plugin 阻塞系统事件** | 长耗时同步操作导致超时 | 复杂操作改为异步或降低 Rank |
| **绕过 Dataverse 安全模型** | 升级后执行失败 | 使用标准安全上下文 |

### 5.3 选项集陷阱（⚠️ 高频大坑）

| 陷阱 | 后果 | 解决方案 |
|------|------|---------|
| **PRD 业务编号 ≠ 选项集实际值** | 状态值错误，JS/Plugin 逻辑异常 | 必须通过实际查询确认选项集值 |
| **某些字段创建与查询值不同** | 创建用 `100000000`，查询返回 `1` | 创建后查询验证，确认实际存储值 |
| **选项集值格式混淆** | 某些字段一致，某些不一致 | 每个选项集字段单独确认 |

**典型案例：**
- PRD 中状态顺序为 1-8（业务编号）
- D365 选项集实际值为 9-16（技术实现值）
- JS 代码初始值设为 1 → 报错 "Invalid value 1 for OptionSetValue"
- 修复：JS 改回实际值，Plugin 触发状态同步调整

### 5.4 升级兼容性陷阱

| 陷阱 | 后果 | 解决方案 |
|------|------|---------|
| **直接修改原生系统组件** | 升级后被重置 | 只扩展自定义组件，不碰原生 |
| **硬编码原生 ID** | 升级后 ID 变更，脚本失效 | 使用逻辑名而非 ID |
| **依赖原生 UI DOM 结构** | 升级后 DOM 重构，JS 失效 | 使用官方 API（formContext） |
| **覆盖原生 Web 资源** | 升级强制还原 | 自定义资源用新文件名 |

### 5.5 BPF 陷阱

| 陷阱 | 后果 | 解决方案 |
|------|------|---------|
| **ConditionJson 无法后端修改** | BPF 阶段与状态值不对应 | 使用 Plugin 同步方案（状态→StageId 映射） |
| **未停用 BPF 就修改 XAML** | 更新失败或数据损坏 | 必须先停用（Draft 状态） |
| **D365 未自动解析新 Plugin Type** | Plugin 注册后不可用 | 手动创建 plugintype 记录 |

## 八、经验教训

### 8.1 成功经验

1. **实体先行策略有效** - 先固化数据模型，再构建上层逻辑
2. **SolutionViewer 价值大** - 可视化确认减少 80% 沟通成本
3. **JSON 配置驱动** - 实体定义可复用、可版本化
4. **快速迭代** - 字段类型调整从小时级降到分钟级
5. **4核心文件体系** - AI 协作标准化，交接成本极低
6. **功能编号追踪** - 功能-测试-Bug 全链路可追溯
7. **C# CLI 工具链** - 绕过 Solution 导入，避免 GUID 冲突

### 8.2 待改进点

1. **Lookup 字段创建复杂** - 需要优化 C# 工具的错误提示
2. **选项集更新不便** - 已有字段不能修改选项值，需特殊处理
3. **字段删除限制多** - 被引用后无法删除，需提前规划命名
4. **BPF ConditionJson 无法后端修改** - 必须依赖 Plugin 同步方案
5. **外部接口文档不完整** - 需建立 Mock 数据机制降低依赖
